# Time-Off Microservice — ExampleHR

A NestJS + SQLite microservice that owns the lifecycle of time-off requests
for ExampleHR while keeping per-employee, per-location balances in sync with
an external HCM system (Workday, SAP, etc).

For the design rationale, alternatives considered, and contract details, see
[`docs/TRD.md`](docs/TRD.md).

---


## Quickstart

```bash
# 1. Install
npm install

# 2. Copy the example env
cp .env.example .env

# 3. Start the HCM mock (port 3100 by default)
npm run start:mock

# 4. In another terminal, start the service (port 3000 by default)
npm run start:dev
```

## Architecture in 30 seconds

- **Ledger + projection** — every balance-affecting event is appended to
  `balance_ledger` (with `hcm_idempotency_key` uniqueness for HCM-driven
  writes); a `balance_snapshots` row per `(employee, location)` materializes
  the current balance and the pending reservation total.
- **Hard reservation** — `available = balance - reserved`. Creating a
  request reserves balance immediately; cancel/reject release it; approve
  attempts a synchronous HCM commit and then converts the reservation to a
  debit.
- **Defensive HCM client** — bounded retries with exponential backoff and a
  circuit breaker; business errors (`INSUFFICIENT_BALANCE`,
  `INVALID_DIMENSION`) are *not* health signals and don't trip the breaker.
- **Outbox** — when HCM is retryably failing, the debit goes to
  `outbox_events`; a polling worker drains pending events; same idempotency
  key on the HCM side prevents double-debits.
- **Reconciliation** — full-corpus push (`POST /api/v1/hcm/batch-ingest`)
  and on-demand admin trigger (`POST /api/v1/admin/reconcile`); both write
  `HCM_SYNC_ADJUST` ledger entries on drift and flag pending requests that
  no longer fit under the new HCM balance.

## REST endpoints

See [`docs/TRD.md` §7](docs/TRD.md#7-api-contract). All write endpoints accept
an `Idempotency-Key` header. All endpoints except `/api/v1/healthz` and
`/api/v1/hcm/batch-ingest` (header-secured) require a `Bearer <jwt>` with
a `role` claim.

## Configuration

All config flows through `apps/timeoff/src/config/config.schema.ts` and is
validated via Joi. See [`.env.example`](.env.example) for the full list.

Notable knobs:

- `DATABASE_SYNCHRONIZE` — true in dev/test (auto-create schema), false in
  prod (use migrations).
- `OUTBOX_WORKER_ENABLED` — turn the background worker on/off (tests run
  it on demand via `POST /api/v1/admin/outbox/drain`).
- `RECONCILE_CRON_ENABLED` — turn the periodic reconciliation cron on/off;
  off by default. The manual endpoint always works.
- `HCM_BATCH_INGEST_SECRET` — symmetric secret for the batch-ingest endpoint.

## Development conventions

- TypeScript strict mode (`strictNullChecks`, `noImplicitAny`).
- Pure domain in `apps/timeoff/src/domain/`. No framework imports there.
- Persistence layer hides TypeORM behind small repository classes that
  accept an `EntityManager` for cross-aggregate transactions.
- Errors split by tier: `DomainError` (4xx user feedback) and `HcmError`
  (4xx/5xx upstream signals). Each has a dedicated exception filter that
  produces `{statusCode, error, message}`.
- Lint: `npm run lint`. Format: `npm run format`.
